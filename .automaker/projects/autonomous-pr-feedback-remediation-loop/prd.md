# PRD: Autonomous PR Feedback Remediation Loop

## Situation
PRFeedbackService already polls every 60s and auto-restarts agents with feedback, CodeRabbitParserService parses structured comments, CodeRabbitResolverService can resolve threads (but only manually), and EMAgent handles approvals. However, 9 critical gaps prevent true autonomy: race conditions between services, no auto-thread-resolution, no CI failure loop, in-memory tracking lost on restart, agents lose context on restart, COMMENTED reviews ignored, no per-thread tracking, no critical evaluation of feedback (blind acceptance), and misleading feature status transitions.

## Problem
When CodeRabbit or a human leaves PR feedback, the system can detect and restart the agent, but the loop isn't deterministic or complete. Agents blindly accept all feedback without evaluation. Review threads stay unresolved even after fixes. CI failures after fixes become dead ends. Tracking state is lost on server restart. The feature status cycles through 'backlog' misleadingly. There's no paper trail showing which feedback was accepted vs denied and why. Two services race to handle the same event causing state corruption. Agents don't see their previous work context when restarted for fixes.

## Approach
Consolidate all feedback orchestration into PRFeedbackService (remove EM duplication). Add feedback triage step: parse each thread, classify actionable vs non-actionable, present to agent for critical evaluation. Agent gets previous context (agent-output.md) PLUS structured per-thread feedback. After agent pushes fixes: auto-run thread resolution to close addressed items, post denial reasons for rejected feedback. Subscribe to pr:ci-failure events and trigger remediation cycle. Persist tracked PRs to disk (extend feature.json). Add pull_request_review webhook for instant detection (keep polling as fallback). Introduce 'remediating' feature status to avoid misleading backlog transitions. Add per-thread tracking to feature metadata. Build antagonistic review capability: agent must justify accepting or denying each feedback item.

## Results
When feedback arrives, the system will: (1) Instantly detect via webhook or 60s polling fallback. (2) Parse into structured per-thread items with severity/location. (3) Load agent's previous context (agent-output.md, executionHistory). (4) Present feedback to agent for critical evaluation. (5) Agent decides per-thread: accept+fix OR deny+reason. (6) Agent implements accepted fixes. (7) Auto-resolve addressed threads, post denial comments with reasoning. (8) Verify CI passes; if fails, trigger another remediation cycle. (9) All decisions logged to feature.json with timestamps and reasoning. (10) No human intervention needed; full audit trail exists. State survives server restarts. No race conditions or corruption.

## Constraints
Must maintain backward compatibility with existing auto-mode flow,Cannot break current EM agent merge workflow for approvals,Must work with existing feature.json schema (extend, don't replace),CodeRabbit GraphQL API rate limits (max 100 threads per query),GitHub webhook signature verification required for security,Agent context from previous run may be very large (handle truncation),Some feedback may be subjective (agent needs clear evaluation criteria),PR iteration limit (currently 2) must remain configurable
