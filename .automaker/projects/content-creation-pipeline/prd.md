# PRD: Content Creation Pipeline

## Situation
Automaker has three production-ready packages — @automaker/flows (LangGraph state graphs, reducers, routing, subgraph composition), @automaker/llm-providers (6-provider abstraction with model categories), and @automaker/observability (Langfuse tracing, prompt versioning, caching). Two sister projects demonstrate proven content creation patterns: Proto Starter uses multi-phase extraction with HITL interrupts and coordinator-to-subagent delegation; MythxEngine uses Send()-based two-wave parallel generation with Langfuse+local fallback prompts. No unified content creation flow exists in Automaker yet.

## Problem
Content creation today is manual or single-threaded: one agent writes one document start-to-finish. This is slow, hard to trace, lacks human feedback gates, produces inconsistent quality, and can't generate structured training data as a byproduct. There's no reusable pipeline for blog posts, technical documentation, training data generation, or HuggingFace dataset creation. The proven patterns from Proto Starter and MythxEngine are siloed in those codebases.

## Approach
Build a content creation pipeline as a LangGraph StateGraph in @automaker/flows with 7 phases: (1) Parallel research via Send() fan-out to web/codebase/existing-content workers; (2) HITL research review gate; (3) Outline planning with section decomposition; (4) HITL outline approval gate; (5) Parallel section generation via Send() to isolated SectionWriter subgraphs with model fallback chains (opus→sonnet→haiku) and Langfuse tracing per generation; (6) Assembly + parallel review (technical accuracy, style, fact-check); (7) HITL final approval then parallel output to markdown, HF JSONL dataset, and metadata. Each phase uses typed state annotations with appropriate reducers (appendReducer for parallel collection, fileReducer for document merging). All LLM calls traced via @automaker/observability. Prompts versioned in Langfuse with local markdown fallback. Content types are Zod-validated schemas. FakeChatModel enables full pipeline testing without API calls.

## Results
A reusable content creation pipeline that: generates blog posts, technical docs, and training data 3-5x faster than sequential writing; traces every LLM call through Langfuse for cost/quality monitoring; supports HITL feedback at 3 gates (research, outline, final); produces HuggingFace-compatible JSONL datasets as a byproduct; handles model failures via fallback chains; resumes from any checkpoint after interruption; works with any provider via @automaker/llm-providers; includes comprehensive docs and examples in the flows package.

## Constraints
Zero coupling to Automaker server code — pipeline lives entirely in @automaker/flows,All LLM calls go through @automaker/llm-providers, never import provider SDKs directly,Tracing via @automaker/observability must be optional (no-op when Langfuse unavailable),Must work with FakeChatModel for testing (no real API calls in test suite),State reducers must handle concurrent Send() updates correctly,HITL interrupts must use LangGraph interrupt() with MemorySaver checkpointing,Output schemas validated with Zod before write,Prompts stored as local markdown with Langfuse versioning overlay,No new package dependencies beyond what @automaker/flows already has,Documentation follows docs/dev/docs-standard.md conventions
