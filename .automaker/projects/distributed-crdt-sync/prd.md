# PRD: Distributed CRDT Sync

## Situation
protoLabs Studio currently operates as a single-instance system where .automaker/ files (features, projects, notes, settings) are local to each machine. The FeatureStore interface, EventBus.broadcast(), and HivemindConfig types are already extracted as seams for multi-instance operation, but no sync layer exists. Each instance maintains its own board state independently. Knowledge (context, memory, skills) syncs via git, but operational state (features, projects) does not propagate between instances.

## Problem
Users with multiple machines (dev laptop, staging VM, production node) or collaborators sharing compute cannot see or act on the same board state. Features created on one instance are invisible to others. Compute sits idle when one instance finishes its backlog while another is overloaded. There is no way to assign tickets across instances, share project ceremonies, or get a unified view of work in progress. This limits the system to single-machine scaling and prevents the hivemind architecture from materializing.

## Approach
Implement a CRDT-based sync layer using Automerge 3 + automerge-repo over WebSocket on Tailscale VPN. Shared data is organized into Automerge documents by domain: features, projects, notes workspace, and shared settings. A proto.config.yaml file (git-tracked) defines shared defaults and instance registry with layered overrides. Sync topology is star with automatic leader election — one instance runs the automerge-repo-sync-server, others connect as WebSocket clients. Cross-instance feature assignment uses a work-stealing protocol where idle instances request features from busy peers, with CRDT field updates propagating assignments conflict-free. The implementation phases in through the existing FeatureStore and EventBus abstractions, ensuring zero breaking changes to callers. Instance-local data (agent output, trajectories, worktrees, credentials) stays local.

## Results
Multiple Automaker instances share board state with sub-100ms propagation. Features created on any instance appear on all others within milliseconds. Cross-instance assignment enables work to flow to available compute automatically. Each instance operates offline-first and merges seamlessly on reconnect. A unified dashboard shows all instances' work. The proto.config.yaml provides shared configuration with per-instance overrides. New data domains can be added by creating new Automerge documents without architectural changes.

## Constraints
Must work over Tailscale VPN (stable IPs, encrypted transport, NAT traversal handled),Must be offline-first — each instance works independently when disconnected,Must not break single-instance operation — CRDT layer is opt-in via proto.config.yaml,Must use existing FeatureStore and EventBus interfaces — no breaking changes to callers,Agent output, trajectories, worktrees, and credentials remain instance-local (never synced),Knowledge layer (context, memory, skills) continues syncing via git — not duplicated in CRDT,Automerge 3 is the CRDT library (JSON-native, 10x memory improvement, automerge-repo networking),Star topology with automatic leader election for 2-10 nodes,proto.config.yaml is git-tracked; instance-local overrides in .automaker/settings.json,Cross-instance assignment via work-stealing protocol, not centralized scheduler,Notes rich-text CRDT (via @automerge/prosemirror) is deferred to Phase 5 — Phase 3 syncs notes at the workspace JSON level
