# PRD: Lifecycle Traceability & Paper Trail

## Situation
protoLabs Studio has three lifecycle systems — Projects (milestones/phases), Features (LE pipeline INTAKE→DONE), and Ceremonies (state machine with standup/retro flows) — connected by an event-driven architecture. Eleven distinct recording mechanisms exist spanning disk persistence (metrics ledger, trajectories, ceremony audit log), external tracing (Langfuse), and in-memory events. The metrics ledger uses append-only JSONL for permanent records. Trajectories survive in a separate directory. The ceremony state machine is a pure transition function persisted to ceremony-state.json. Feature status changes are tracked in statusHistory[] on feature.json. The CompletionDetectorService cascades feature→epic→milestone→project completion via events.

## Problem
Five critical gaps break the lifecycle chain: (1) milestoneSlug is NEVER set on features during scaffolding — both orchestrateProjectFeatures() and ProjectService.createFeaturesFromProject() omit it, breaking the entire CompletionDetectorService milestone→project cascade and all ceremonies that depend on milestone:completed and project:completed events. (2) The event system is entirely fire-and-forget — no event log, no audit trail, no replay capability. Critical lifecycle events vanish after delivery. (3) ArchivalService deletes the entire feature directory after 2 hours including statusHistory, executionHistory, remediationHistory, agent-output.md, and handoffs — destroying the paper trail. (4) The metrics ledger only records completed features — failed/escalated/abandoned features have zero permanent record. (5) Failure classifications from FailureClassifierService are ephemeral — the structured analysis (category, confidence, recovery strategy) is consumed and discarded, only the raw error string persists as statusChangeReason.

## Approach
Seven milestones in strict dependency order, TDD-first throughout. M1: Test Foundation — write regression tests for CompletionDetectorService cascade, archival behavior, event emission patterns, and ledger recording BEFORE changing any production code. M2: Fix Milestone Cascade — set milestoneSlug and phaseSlug on features during scaffolding in both creation paths, fix CompletionDetectorService to cascade correctly, verify end-to-end with M1 tests. M3: Event Persistence Layer — append-only JSONL event ledger capturing all lifecycle events with correlation IDs (projectSlug, featureId, milestoneSlug, ceremonyId). M4: Archival Reform — archive feature data to .automaker/archive/{featureId}/ instead of deleting, preserve feature.json and agent-output.md permanently, add retention policy config. M5: Expanded Ledger — record all terminal states (done, escalated, abandoned) in metrics ledger, persist failure classifications on feature.json, store all Langfuse traceIds as array. M6: Automatic Status Events — featureLoader.update() auto-emits feature:status-changed when status changes, remove manual emission from callers, ensure all downstream listeners fire consistently. M7: Traceability Enrichment — persist LE session ruleLog snapshots, restore PR tracking state on restart from feature.json, add reverse lookup fields for full bidirectional tracing.

## Results
Every lifecycle transition produces a permanent, queryable record. The milestone→project completion cascade works automatically, triggering ceremonies. Feature history survives archival in .automaker/archive/. Failed features appear in the ledger alongside successes. A unified event ledger captures all lifecycle events with correlation IDs for cross-system tracing. Any feature can be traced bidirectionally: feature→phase→milestone→project and project→milestone→phase→feature. The entire system has regression test coverage validating the paper trail at each seam.

## Constraints
TDD-first: write tests for each fix BEFORE implementing the fix — M1 is entirely tests,Each milestone must be independently deployable — no big-bang rewrites,Use the existing append-only JSONL pattern (ledger-service.ts) for new persistence layers,Do not change the Feature type shape in breaking ways — new fields must be optional,Archive data must be queryable — not just a dump directory,Event persistence must be fire-and-forget (like trajectory writes) — never block the caller,Do not break existing CeremonyService event subscriptions or CeremonyAuditLogService,Preserve backward compatibility with features that lack milestoneSlug (existing data),featureLoader.update() status event emission must be idempotent — no double-fires,All new services must follow the existing dependency injection pattern via wiring.ts
